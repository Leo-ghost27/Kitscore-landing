## The investigation, and what it actually found

**Claim: commit `887d6a4` was pushed successfully this session.** False. Checked `origin/main`, the other branch mentioned in that doc (`app/creator-sponsor-mvp`), and reflog — that commit doesn't exist anywhere. The push likely failed silently or never completed, despite what that session believed. Not consequential either way: the admin Signups panel it describes is already live via a different, independent commit from an earlier session (`ea6e8d1`).

**Claim: "something was deployed to Vercel outside of a normal git push" because `fn_get_my_profile()` doesn't exist in the repo.** This is a misdiagnosis, not a real finding. Investigated properly:
- `fn_get_my_profile()` absolutely exists — I created it myself, two sessions ago, as part of the `profiles.email` leak fix. Confirmed still intact.
- The client code that *calls* it (`app/supabase-client.js`'s `getCurrentProfile()`) has been in the repo since that same session's commit.
- What genuinely doesn't exist anywhere in git: **any SQL migration file, at all, in this project's entire history.** Checked — zero `.sql` files have ever been committed. Every database change across every session (mine included) has been applied directly to Supabase and never version-controlled.

**The real explanation:** this was never a mystery deploy bypassing git. It's that the database schema itself has never been tracked in source control by any session on this project, from the start. A function existing in Supabase but not in a git-tracked migration file is the expected result of that practice, not evidence of drift between what git says and what Vercel is running. I found zero evidence of actual frontend/API code reaching production outside of a normal push.

**One thing I couldn't fully verify either way:** the "polling `fn_get_my_profile()` 4x/second indefinitely" observation from Supabase's API logs. I don't have log access to confirm the exact pattern, but I checked every client file for any retry/polling loop and found none — `getCurrentProfile()` is called once per page load, normally. My best explanation: a stuck user (Halima, per that session's own writeup) refreshing a hung page repeatedly is ordinary human behavior that would produce exactly this signature in aggregate logs over a short window, without any actual code bug causing an infinite loop. Not fully provable from here, but nothing in the current codebase supports "phantom mystery code" as the explanation.
