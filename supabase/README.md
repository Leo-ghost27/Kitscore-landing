# Supabase schema tracking

## Why this folder exists (read this first)

As of July 4, 2026, this repository had **zero SQL migration files, ever, in its entire git history** â€” every database change (functions, triggers, RLS policies, grants) across every session on this project was applied directly to Supabase, and never committed to git. This was flagged by one session as "git/production drift" â€” the theory being that Vercel was somehow running code that didn't come from a `git push`.

That specific theory doesn't hold up: there's no evidence anywhere (git history, branches, reflog) of frontend/API code reaching production outside of a normal push. What's actually true is narrower and less alarming: **the database schema itself was never version-controlled at all**, by any session, from the start of this project. A function existing in Supabase but not in the git repo isn't a mystery deploy â€” it's the expected result of applying SQL directly via the Supabase dashboard/API instead of committing a migration file, which is what every session (including this one) has been doing.

`schema-baseline-2026-07-04.sql` is a full, verified snapshot of every custom function, trigger, and RLS policy that exists in the database as of this date â€” generated by querying Postgres's own catalogs (`pg_proc`, `pg_trigger`, `pg_policy`) directly, not reconstructed from memory or old commit messages. It's a **reference snapshot**, not something meant to be re-run â€” running it against the live database would just recreate what's already there (most statements use `CREATE OR REPLACE` or `IF NOT EXISTS` patterns where the original did, but this wasn't written as an idempotent migration script, it's a documentation snapshot).

## The practice going forward

Any future database change â€” a new function, trigger, RLS policy, or grant â€” should be added as a new dated file in this folder (`YYYY-MM-DD-short-description.sql`), committed in the same push as any related application code. This is the only way to keep git and the database schema from silently diverging again.

## Before recreating any function, check the baseline first

On 2026-07-15, two parallel sessions each ran a `DROP FUNCTION` +
`CREATE FUNCTION` on `fn_get_evekit_profile` within hours of each
other. Each session had a correct-at-the-time picture of the function,
but neither checked what the *other* had just changed â€” so the second
session's version silently reverted a column (`view_count`) the first
had just added. It was caught by luck, on a manual audit, not by any
process.

**If you are about to `DROP`/`CREATE` (or otherwise fully rewrite) an
existing function, trigger, or policy: check its current definition in
`schema-baseline-*.sql` (the most recent dated file in this folder) or
directly against live Postgres (`pg_get_functiondef`) first â€” not just
what you remember writing, and not just what's live *right now* from a
query, since another session may have a change in flight that hasn't
been reflected anywhere you'd see it. Diff your intended change against
that, not against a spec or memory of an earlier version.** This is
the single highest-leverage check to prevent silent regressions when
more than one session may be touching the schema.
